I've been working a lot on the Yumma CSS documentation lately, trying to make sure that all the information presented is accurate and clear, without creating unnecessary confusion or complication in the process of digesting the content.

{/* truncate */}

This guideline is intended to demonstrate the standards and practices I use to maintain and ensure that the Yumma CSS documentation remains clear, consistent, and easy to understand.

## Why do I care?

The mission of this documentation is to provide accurate and clear information about Yumma CSS in general, to help advanced or novice users effectively use the framework in their projects.

## The structure

I used to organize Yumma CSS docs by categories and subcategories, the problem is that in the long run maintaining and updating the documentation will only make it harder to maintain.

### Previous structure

```
Box Model
├─ Box Sizing
├─ Dimension
│  ├─ Dimension
│  ├─ Max Dimension
│  └─ Min Dimension
├─ Height
│  ├─ Height
│  ├─ Max Height
│  └─ Min Height
├─ Margin
│  ├─ Margin Bottom
│  ├─ Margin Left
│  ├─ Margin Right
│  ├─ Margin Top
│  ├─ Margin X
│  ├─ Margin Y
│  └─ Margin
├─ Padding
│  ├─ Padding Bottom
│  ├─ Padding Left
│  ├─ Padding Right
│  ├─ Padding Top
│  ├─ Padding X
│  ├─ Padding Y
│  └─ Padding
└─ Width
   ├─ Max Width
   ├─ Min Width
   └─ Width
```

So I decided to adopt documenting related topics on the same page, this way I add more value to the content present without distracting users with multiple pages.

### Updated structure

```
Box Model
├─ Box Sizing
├─ Dimension
├─ Height
├─ Margin
├─ Padding
└─ Width
```

 Not to mention that it helps keep the codebase clean and allows the Algolia crawler to index the pages faster and more efficiently.

- [x] **Main categories**: Use main categories to group related topics.
- [x] **Subcategories**: Create subcategories if a category has more than three pages.
